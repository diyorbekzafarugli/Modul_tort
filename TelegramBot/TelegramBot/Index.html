1. Dastur Fayllari va Ularning Vazifasi (Arxitektura)

Loyiha Clean Architecture (Toza arxitektura) tamoyillariga asoslangan bo'lib, vazifalar qat'iy taqsimlangan. U uchta asosiy ustunga tayanadi.

A) Program.cs (Dasturning poydevori)

Dastur ishga tushganda faqat 1 marta ishlaydi. Uning asosiy vazifasi – Dependency Injection (DI) qutisini kerakli asboblar bilan to'ldirish.

  • AddDbContext: Ma'lumotlar bazasini ulaydi. Birinchi ulanishining sababi — boshqa xizmatlar bunga qaram.
  • AddHttpClient: Tashqi API'larga chiqish uchun internet resurslarini tejovchi zavod yaratadi.
  • AddScoped<BotHandler>(): Har bir yangi xabar uchun alohida, izolyatsiya qilingan muhit yaratadi.
  • AddHostedService<BotHostedService>(): Botni 24/7 orqa fonda ishlashga majbur qiladigan xizmat.

B) BotHostedService.cs (Mator va Qorovul)

Dastur yopilmaguncha uzluksiz ishlaydigan mator (Singleton).

Arxitekturaviy yechim: Telegramdan xabar kelganda u o'zi javob bermaydi. U CreateScope() orqali yangi DI "muhit" ochib, yangi BotHandler ga ishni topshiradi. Bu orqali Minglab xabarlar kelganda dastur qotib qolmasligi va Memory Leak (xotira to'lib qolishi) oldi olinadi.

C) BotHandler.cs (Miya va Mantiq)

Botga xabar kelganidagina yashin tezligida ishlab, o'chib ketadigan qism (Scoped).

2. Algoritm — "Finite State Machine" (FSM)

Ro'yxatdan o'tish qismida nima uchun switch (user.State) ishlatdik?

Telegram API "Stateless" (Xotirasiz) hisoblanadi. Bot foydalanuvchining 5 daqiqa oldin yozgan xabarini o'z-o'zidan eslay olmaydi. Buni hal qilish uchun Chekli Holatlar Mashinasi (FSM) qo'llaniladi. Biz foydalanuvchining qaysi bekatda to'xtaganini bazaga yozib qo'yamiz. Tizim o'chib yonsa ham, bot algoritmni kerakli joydan davom ettiradi.

3. Paketlar va Ularning Vazifalari (Dependencies)

Loyihada o'rnatilgan paketlar tasodifiy emas, ular .NET ekotizimining eng kuchli asboblaridir:

Paket Nomi Ishlab chiquvchi Qanday ishlaydi va Imkoniyatlari
Microsoft.Extensions.Hosting Microsoft DI (Dependency Injection) qutisini va orqa fon vazifalarini yaratadi. Klasslarni bir-biriga bo'sh bog'lash (Loose coupling) imkonini beradi.
EF Core.Sqlite & Tools Microsoft ORM texnologiyasi. SQL yozish o'rniga, C# metodlari yoziladi va u orqa fonda SQL ga o'giriladi. Tools paketi migratsiya qilish uchun javobgar.
Telegram.Bot Hamjamiyat Telegramning murakkab HTTP (REST) so'rovlarini tayyor va qulay C# klasslariga (masalan: SendMessageAsync) o'girib beradi.

4. Katta Platformalarga Integratsiya (Pro Daraja)

Agar bot yirik loyihaning (Ijtimoiy tarmoq, O'quv platformasi) bir qismi bo'lsa, uni quyidagi professional paketlar bilan kengaytirish shart:

  • Ma'lumotlar Bazasi: SQLite o'rniga PostgreSQL o'rnatiladi. U millionlab tranzaksiyalarni oson ko'taradi.
  • State Management (Tezkor xotira): FSM (Holatlar) ni bazada saqlash tizimni sekinlashtiradi. O'rniga Redis ishlatiladi (RAM darajasida ishlaydi).
  • Toza Arxitektura (CQRS): Katta switch lar o'rniga MediatR paketi qo'yiladi. Har bir komanda alohida faylda ishlaydi.
  • Loglash tizimi: Console o'rniga Serilog ishlatilib, xatolar fayllarga yoki Seq/Elasticsearch ga yoziladi.

5. Bot Yaratish Turlari (Arxitektura farqi)

📡 Long Polling

Bot har soniyada Telegramdan "Xabar bormi?" deb so'raydi. (Bizda hozir shu).

Plyus: Localhost'da test qilish oson.

Minus: Server resurslarini ko'p yeydi.

🚀 WebHook (Pro)

Bot jim turadi. Telegram xabar kelganda to'g'ridan-to'g'ri sizning Server API'ngizga jo'natadi.

Plyus: Resurslarni tejaydi, millionlab odamga dosh beradi.

Minus: SSL sertifikatli ochiq domen kerak.

6. UI (Interfeys) Evolyutsiyasi

Telegramda bot interfeysini professional qilishning 3 bosqichi mavjud:

  1. Reply Keyboard (Oddiy klaviatura): Ekran pastidagi tugmalar. Asosan navigatsiya uchun.
  2. Inline Keyboard (Shaffof tugmalar): Xabar ostiga yopishadigan tugmalar. Ular bosilganda xabarning o'zi tahrirlanadi (EditMessageText). Bu chinakam professional yondashuv.
  3. Telegram Web Apps (TWA): Bot ichida ochiladigan haqiqiy HTML/React veb-saytlar (Notcoin, Blum). Bot faqatgina tizimga kirish (eshik) vazifasini o'taydi.
